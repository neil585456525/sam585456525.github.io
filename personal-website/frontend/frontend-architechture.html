# 前端軟體架構選擇

# 一、Web 前端服務架構

在制定 Web 前端架構時，須考慮需求、開發速度以及後續的可維護性，每個不同的架構也不是絕對二分的方式，它更像是一個光譜，需要根據每個團隊的需要，去選擇或客製化最適合自己的方式。

# Traditional SSR（Server Side Render）

  • 接受請求後 controller 組合資料並拼接 HTML 字串
  • 簡單直接，部屬方式單純，但開發如後台等有複雜交互與狀態的應用程式級系統，會導致前端架構複雜

適合靜態網頁如官網、或互動較簡單之應用。

後端程式語言輸出 HTML 結構 + Jquery/Javascript 執行部分動畫

# SPA（Single Page Application）

  • 在瀏覽器中使用 js 生成所有 HTML 內容
  • 大量使用 Ajax 進行資料溝通
  • 更穩定前端軟體架構（組件化、狀態管理、資料驅動畫面）
  • 通常需要MVVM 框架、前端編譯軟體
  • SEO 差、首頁渲染較慢

適合操作複雜的後台管理系統。

React、Vue、Angular、Svelt

# Isomorphic SSR

  • 同構 Web 應用，將新舊兩種方式混和
  • 由 Nodejs 引擎預渲染前端軟體，並進行注水與脫水
  • 解決 SEO 與加速首屏渲染，但會消耗更多的伺服器資源（快取、預渲染）

適合複雜但重視 SEO 的內容應用，如論壇、社交軟體、現代官網。

Next、Nuxt、Universal Angular

# 二、基於 Web 技術的跨平台架構

一般會使用跨平台技術是為了減少 Web 開發人員的學習成本，並盡量能做到 UI 畫面與互動邏輯的復用。

通常關注於螢幕尺寸與運行環境兩種，螢幕尺寸通常可分為：桌上型電腦、平板電腦、手機，運行環境則分為：瀏覽器引擎與裝置原生作業系統兩種。

# 跨螢幕尺寸解決方案

# 多獨立應用架構

最傳統的解決方案，直接製作手機版、電腦版、平板版等等不同裝置使用的畫面（HTML），並且判斷使用者裝置導向到不同的應用程式，將導致高昂的維護成本，也難以共用畫面上的邏輯。

# RWD 響應式畫面設計

主要利用 CSS Media Queries 屬性，可以針對不同的螢幕寬度，應用不同的 Style 內容，以求在 HTML 上的最大復用。 局部利用 Js 去切換專屬於各裝置的部分組件，去完成跨裝置間差異更大的畫面，目前已成為業界主流的解決方案。

# 跨裝置解決方案

RWD 可以初步解決跨螢幕尺寸的顯示問題，可是難以突破瀏覽器 API 的限制，如直接操作電腦中的檔案文件，或是使用手機的定位、串流等原生的 API。並且也無法在脫離瀏覽器環境的情況下執行，將面臨無法上架 APP Store 的窘境。

# 包裝 Web 渲染引擎

  • 基於同瀏覽器環境的 Web 渲染引擎，並且封裝原生系統 API
  • 相較簡易的開發環境，並且可以相容於多數 Web 前端應用
  • 在快速滾動、影片與模型渲染等場域可能會因為手機性能而出現卡頓

Cordova（手機）、Electron（電腦 -> Vscode）

# 封裝原生裝置渲染引擎

  • 接受 HTML、CSS 等瀏覽器語言，並且直接呼叫原生的選染引擎進行渲染
  • 會限制部分框架
  • 接近原生應用的渲染效能

React Native、NativeScript

# 其他解決方案

# PWA（漸進 Web APP）

基於純粹的 WEB 技術，加上了 Service Worker 來處理快取軟體

# Flutter

Google 推出的跨 Android、ios 雙平台的 UI 解決方案，並且正在開發編譯成 Web 應用
